# PHASE 1 SUMMARY — 要件定義・アーキテクチャ設計

> 担当: Architect Agent
> ステータス: 設計確定（一部、ユーザー確認待ち項目あり → 「§9 要確認事項」参照）
> 最終更新: 2026-06-21

現場で撮影した写真を「会社名（撮影地点）／案件名」ごとに自動仕分けし、帰社時に Windows PC のローカルフォルダへ自動格納するシステムの、全体設計と各種仕様を確定する。

---

## 1. システム全体像

```
[現場・オフライン]                          [帰社後・同一Wi-Fi（LAN）]
┌─────────────────────────┐                ┌──────────────────────────────┐
│  スマートフォン (PWA)      │                │  Windows PC (常駐サーバー)        │
│                          │                │                              │
│  案件名入力               │                │  ① mDNS でPCを広告              │
│  → カメラ撮影             │   帰社しLAN     │  ② HTTPSで写真を受信            │
│  → GPS自動取得           │  ───────────►  │  ③ Places APIで会社名解決       │
│  → メタJSON付きで         │   転送ボタン    │  ④ フォルダ自動生成              │
│    端末内キュー(IndexedDB) │   1アクション   │  ⑤ ファイル格納                 │
│    に保存（完全オフライン） │                │  ⑥ 完了応答                    │
└─────────────────────────┘                └──────────────────────────────┘
```

- **現場では完全オフライン動作**：撮影・GPS・案件名の付与をすべて端末内で完結させ、写真とメタデータを IndexedDB の送信キューに貯める。電波の有無に依存しない。
- **帰社後（同一Wi-Fi）にのみ通信**：PWA が LAN 上の PC を自動検出し、HTTPS でキュー内の写真を一括送信。送信成功分はキューから削除。
- **会社名の解決は PC 側で実施**：Google Maps の API 呼び出しは PC に集約（APIキーを端末に置かない／課金を一元管理／逆ジオコーディングロジックの保守を一箇所に）。

## 2. 技術スタック確定

| 層 | 採用技術 | 理由 |
|---|---|---|
| PWA フロント | Vanilla HTML / CSS / JavaScript（ビルドツール無し） | 要件指定。低依存・保守性。 |
| PWA オフライン保存 | IndexedDB（画像Blob＋メタJSON） | localStorageは容量・Blob非対応で不可。写真キューに必須。 |
| PWA オフライン動作 | Service Worker（アプリシェルをキャッシュ） | 現場でのオフライン起動・インストール性確保。 |
| PWA 配信元 | GitHub Pages（HTTPS・正規証明書） | **推奨**。camera/GPSに必須の secure context を無償で確保。更新配布が容易。代替案は§4参照。 |
| PC 側サーバー | **Python 3.11+ / FastAPI + Uvicorn** | **推奨**。EXIF/画像処理ライブラリ（Pillow, piexif）が成熟。requestsでMaps API。非同期受信に強い。 |
| PC 常駐化 | Windows: バッチ/PowerShell起動 → 将来 NSSM 等でサービス化 | 段階的。MVPは起動スクリプト＋タスクトレイ常駐から。 |
| サービス検出 | mDNS / Zeroconf（`zeroconf` パッケージ）でホスト名を広告 | 同一Wi-Fi内で固定ホスト名（例 `genba-pc.local`）でPC発見。IP直打ち不要。 |
| 逆ジオコーディング | **【MVP確定】住所ベース仕分け（逆ジオコーディングで住所/地名を取得）。Places APIによる会社名解決は将来オプション** | §3参照。ユーザー判断によりMVPは住所ベースで開始。 |
| ローカル通信 | HTTPS（PC側 自己署名証明書）＋ CORS | secure context間通信を成立させる（§4）。 |

> Pythonを推奨とするが、Node.js(Express)でも実装可能。**§9でユーザー最終確認。**

## 3. 撮影地点の解決ロジック（MVP=住所ベース、ユーザー判断で確定）

要件は「GPS逆ジオコーディングで会社名を取得」だが、**標準の逆ジオコーディングが返すのは住所であって会社名ではない**。座標から事業者名を得るには有料の Places API が必要となるため、**ユーザー判断により、MVPでは「住所（地名）ベースの仕分け」で開始する**。会社名の自動解決（Places API）はフェーズ後半またはv2の拡張オプションとして設計上の拡張点を残す。

**【MVP】解決パイプライン（PC側）:**

1. **逆ジオコーディングで住所を取得** → トップ階層のフォルダ名に代表地名を採用（例: `新宿区西新宿2丁目`）。粒度（市区町村／丁目）は設定で調整可とする。
2. **取得失敗／APIエラー**：固定フォルダ `_未分類_GPSなし` に格納（データは絶対に失わない）。
3. **名称ゆれ・課金抑制**：座標を量子化（小数4桁≒約11mグリッド）したキー → 確定フォルダ名 の対応表（`location_cache.json`）をPC側に保持。一度解決した地点を再利用し、**人が後から「正式な会社名」へ手修正できる**（フォルダ名を書き換えるだけで以降その地点は自動で同名へ）。

**逆ジオコーディングの実装選択（コスト方針）:**
- 「住所ベース」でも Google **Geocoding API は従量課金**である点に注意（ただし逆ジオコーディングは安価で、Googleの月次無料クレジット枠内に小規模利用なら収まりやすい）。
- 完全無料を優先する場合の代替: **OpenStreetMap / Nominatim**（無料の逆ジオコーディング。ただし利用規約上のレート制限（≒1req/秒）・大量/商用利用の配慮が必要）。→ **§9-3 で最終選択。** いずれを選んでも `location_cache.json` でAPI呼び出し回数を最小化する。

**【将来オプション / v2】会社名の自動解決:**
- Places API (Nearby Search) を上記パイプラインの先頭に差し込み、半径 `R`（50m→100m→200m）で事業者名を検索 → 該当時は会社名フォルダ、非該当時は住所フォルダにフォールバック。
- 現場（建設現場・空き地等）はPOI未登録が多く自動精度に限界があるため、`location_cache.json` の人手修正運用は会社名モードでも前提とする。
- 有効化にはGCPプロジェクト・課金・Places API キーが必要。設計上は「解決プロバイダ」を差し替え可能なインターフェースにしておく。

## 4. ローカルネットワーク通信プロトコル設計

### 4.1 secure context 問題と解決方針

PWAのカメラ(`getUserMedia`)・GPS(`geolocation`)・Service Workerは **secure context（HTTPSまたはlocalhost）** でしか動作しない。一方、帰社時の送信先はLAN内PC。ここが本システム最大の技術的ハードル。

**採用方針:**
- PWA本体は **GitHub Pages（正規HTTPS）** で配信 → camera/GPS/インストールは確実に動作。
- PC側サーバーは **HTTPS（自己署名証明書）** で待受。HTTPSページ→HTTPSエンドポイントなので mixed-content にならない。
- 初回のみ、スマホに **自己署名証明書（ルートCA）を信頼インストール**（iOS: 構成プロファイル、Android: CA証明書インストール）。これは「クラウド不使用・ローカル完結」を満たすための不可避な一度きりのセットアップコスト → フェーズ6のセットアップガイドで手順化。
- PC側はCORSで PWA配信オリジン（GitHub Pages のURL）を許可。

> 代替案（自己署名を避けたい場合）: PWAもPCサーバーから配信し localhost ではなく同一自己署名オリジンに統一する案があるが、結局スマホでの証明書信頼は必要。GitHub Pages 配信のほうが配布・更新が容易なため推奨とする。**§9で最終確認。**

### 4.2 サービス検出（自動ディスカバリー）

- PC側が mDNS で `genba-photo._http._tcp.local`（仮）を広告し、固定ホスト名 `genba-pc.local` を解決可能にする。
- PWAは設定保存済みのホスト名/URLへ接続を試行。失敗時は手動でIP入力できるフォールバックUIを用意。
- （ブラウザからの直接mDNSクエリは不可のため、PWA側は「保存されたPCアドレスへ接続」＋「QR/手入力で初回登録」方式とする。）

### 4.3 転送フロー（エンドツーエンド）

```
PWA                                   PC (FastAPI / HTTPS)
 │  GET  /api/ping                      │  → {status:"ok", version, free_space}
 │ ───────────────────────────────────►│
 │  POST /api/photos  (1枚ずつ or バッチ) │
 │   multipart: file + meta(JSON)       │  → EXIF確認・会社名解決・格納
 │ ───────────────────────────────────►│  → {status, saved_path, dedup}
 │  ← 成功した photo_id をキューから削除   │
 │  （全件完了まで順次／リトライ）          │
```

## 5. メタデータ形式定義

PWAが各写真に付与し、送信時に同梱するJSON。**写真Blobとは別にJSONで送る**（EXIF GPS埋め込みは端末差が大きく信頼性が低いため、正本はこのJSONとする。EXIFへの併記は任意）。

```jsonc
{
  "schema_version": 1,
  "photo_id": "uuid-v4",            // 端末生成。重複排除キー
  "project_name": "○○ビル改修工事",  // 案件名。アプリ非経由時はnull
  "captured_at": "2026-06-21T09:32:11+09:00", // ISO8601・端末ローカル時刻
  "source": "app",                   // "app" | "native"(未案件扱い)
  "gps": {
    "lat": 35.689487,
    "lng": 139.691706,
    "accuracy_m": 8.0,               // 取得精度(m)。null可
    "captured_at": "2026-06-21T09:32:10+09:00",
    "available": true                // false=GPS取得失敗
  },
  "device": { "platform": "iOS", "ua": "..." },
  "original_filename": "IMG_0001.jpg",
  "mime": "image/jpeg"
}
```

- `source:"native"`（アプリ非経由・端末カメラ撮影分）は案件名なし → 後述の `未案件` フォルダへ。
- `gps.available:false` → `_未分類_GPSなし` フォルダへ。

## 6. フォルダ命名規則・格納仕様

### 6.1 フォルダ構造

```
[指定ルート（既定: マイドキュメント\現場写真）]
├── [地点フォルダ＝住所/地名（MVP）。将来は会社名に手修正/自動解決可]
│   ├── [案件名]
│   │   └── <captured_at>_<photo_id短縮>.jpg
│   ├── [別の案件名]
│   │   └── ...
│   └── 未案件                         ← source=native（案件名なし・GPSあり）
│       └── ...
├── [別の地点フォルダ]
│   └── ...
└── _未分類_GPSなし                    ← GPS取得失敗分（地点解決不能）
    └── ...
```

> MVPの第1階層は「住所/地名」フォルダ。`location_cache.json` の名前を会社名へ書き換えれば、以降その地点は会社名フォルダへ自動格納される（将来のPlaces API自動解決への移行もこの層で吸収）。

### 6.2 命名ルール

- **Windows禁止文字** `\ / : * ? " < > |` および制御文字は全角化 or `_` 置換。前後の空白・末尾ピリオド除去。予約名(CON, PRN等)回避。
- 会社名・案件名は最大長を制限（例: 各80文字）。
- ファイル名: `YYYYMMDD_HHMMSS_<photo_id先頭8桁>.<ext>`（時系列ソート可能・衝突回避）。
- **重複排除**: `photo_id` を基準に、既に同一IDが格納済みならスキップ（再送・二重送信に強い）。内容ハッシュ(SHA-256)も補助的に記録。
- **文字コード**: フォルダ/ファイル名はUTF-8、PythonでWindowsパス安全に処理。

## 7. PC側API仕様（フェーズ3/4で実装）

| Method | Path | 用途 | リクエスト | レスポンス |
|---|---|---|---|---|
| GET | `/api/ping` | 死活・バージョン・空き容量確認 | — | `{status, version, root_path, free_space_mb}` |
| POST | `/api/photos` | 写真1枚＋メタを受信・格納 | `multipart/form-data`: `file`(画像), `meta`(JSON文字列) | `{status, photo_id, saved_path, company, project, dedup:bool}` |
| GET | `/api/config` | 現在の保存ルート等取得 | — | `{root_path, search_radius_m, ...}` |
| POST | `/api/config` | 保存ルート・検索半径等の更新 | `{root_path?, search_radius_m?}` | `{status, config}` |

- 認証: LAN前提だが、初回ペアリングで発行する共有トークンを `Authorization` ヘッダに付与（軽量・なりすまし防止）。
- エラー時もHTTPステータス＋`{status:"error", reason}` を返し、PWA側はキュー保持＆リトライ。

## 8. エッジケース方針（詳細検証はフェーズ5）

| ケース | 方針 |
|---|---|
| GPS取得失敗 | `gps.available:false` → `_未分類_GPSなし` へ格納。データは失わない。 |
| ネット切断/転送中断 | IndexedDBキューに保持。成功分のみ削除し再開時に続きを送信。 |
| 重複ファイル/再送 | `photo_id` で冪等化。既存ならスキップして成功扱い。 |
| 逆ジオコーディング失敗/キー失効 | 地点未解決で `_未分類_GPSなし` 配下の `_地点未解決_<座標キー>` へ格納し、ログ警告。データは失わない。 |
| 地点名の表記ゆれ | `location_cache.json` で座標→確定名を再利用。人手修正可（会社名への書き換えもここで）。 |
| （v2）Places API該当無し | 半径拡大→住所フォルダ→座標キーフォルダの順でフォールバック。 |
| 大量・大容量写真 | 1枚ずつストリーミング受信＋逐次キュー削除でメモリ抑制。 |
| PCディスク満杯 | `/api/ping` の空き容量で事前検知し、PWAに警告表示。 |
